專案介紹

本專案主題為「校園 AI 問答助手」，目標是透過 AI 問答、FAQ 知識庫、截止日提醒與錯答回報等功能， 協助學生快速取得正確的校務資訊，同時降低行政人員處理重複問題的負擔。

  • 姓名：張同學（虛構）

功能優先級（MVP）清單

依據 Persona：新生、在校生、行政人員

優先級定義：P0 必做 / P1 應做 / P2 可做

P0（必做）

1) 校務問答（RAG + 來源引用）

功能目標：提供可追溯、可驗證的校務問答

對應痛點：資訊分散、版本不明、回答信任不足

核心需求

  • 自然語言提問
  • 回答附來源連結與更新時間
  • 無答案時提供轉人工/相關單位

驗收標準（Acceptance Criteria）

  • 提問後 3 秒內回應（P95）
  • 80% 回答含至少 1 個有效來源
  • 無答案情境皆有替代建議

KPI

  • 首次提問解決率（FTR）≥ 60%
  • 平均找答案時間（TTA）下降 30%
2) 知識庫後台（FAQ/公告匯入與版本管理）

功能目標：讓行政端可快速更新與治理內容

對應痛點：更新成本高、口徑不一致

核心需求

  • FAQ 新增/編輯/下架
  • 版本號與生效時間
  • 權限分級（管理員/編輯者）

驗收標準

  • 後台可在 5 分鐘內完成一則 FAQ 更新並發布
  • 每次更新保留版本紀錄與操作人

KPI

  • 錯誤資訊修正時間（MTTR）≤ 1 工作天
3) 截止日與重要事項提醒

功能目標：降低學生錯過期限

對應痛點：公告冗長、期限易漏看

核心需求

  • 問答中可辨識截止資訊
  • 設定提醒（App 內/Email）
  • 個人待辦清單

驗收標準

  • 使用者可一鍵把回答轉為提醒
  • 提醒可編輯時間與頻率

KPI

  • 期限類問題重複詢問率下降 20%
4) 回答回饋與錯答回報

功能目標：建立持續優化閉環

對應痛點：錯答修正慢

核心需求

  • 有幫助/沒幫助評分
  • 錯誤回報（分類 + 補充）
  • 管理端回報佇列

驗收標準

  • 每則回答皆可回饋
  • 回報可追蹤狀態（已受理/修正中/已完成）

KPI

  • 錯答回報處理 SLA：48 小時內

P1（應做）

5) 個人化首頁（身分/年級/系所）

功能目標：提升資訊命中率

對應痛點：跨平台反覆查詢

  • 可依身分顯示不同快捷入口與熱門問題
  • 首頁導流到有效答案率 ≥ 50%

6) 回答轉任務（To-do）

功能目標：讓資訊可直接執行

對應痛點：知道規則但無法落地行動

  • 回答可拆解為步驟清單並勾選完成
  • 任務建立後 7 日完成率 ≥ 40%

7) 行政儀表板（熱門問題/時段/單位）

功能目標：支援公告與流程優化

對應痛點：難掌握服務瓶頸

  • 可查看 Top 問題、趨勢、錯答熱點
  • 重複性問題量月降 15%

P2（可做）

8) 多輪情境引導

功能目標：降低複雜流程理解成本

驗收標準：提供分步引導與進度保存

9) 跨系統進度整合

功能目標：減少來回查詢

驗收標準：可在同一頁查看主要流程狀態

10) 高峰期智能分流策略

功能目標：在選課季維持服務品質

驗收標準：問題可自動分流至模板/人工窗口

建議 Sprint 切分（6 週）

  • Sprint 1（第 1–2 週）：P0-1 問答核心、P0-2 知識庫基礎
  • Sprint 2（第 3–4 週）：P0-3 提醒、P0-4 回饋閉環
  • Sprint 3（第 5–6 週）：P1-5 個人化、P1-7 儀表板基礎

上線門檻（Go/No-Go）

  • P0 功能完整可用且通過 UAT
  • 主要查詢情境（註冊/選課/請假）可被正確回覆
  • 錯答回報流程可在 SLA 內關單
  • 監控指標可於後台即時查看

二、詳細項目（User Story + 驗收條件）

PB-001｜校務自然語言問答（含來源引用）【P0】

User Story

As a 在校學生,
I want 使用自然語言詢問校務問題並看到來源,
so that 我能快速取得可信答案並立即行動。

Acceptance Criteria

  1. 使用者輸入問題後，系統在 3 秒內回覆（P95）。
  2. 回答需顯示至少 1 個來源（網址或文件）與更新時間。
  3. 若無法回答，需提供替代建議（相關單位、關鍵字、人工窗口）。
  4. 可記錄提問、回答、點擊來源事件供後續分析。
PB-002｜FAQ/公告知識庫後台（CRUD + 版本）【P0】

User Story

As an 行政人員,
I want 維護 FAQ 與公告版本,
so that 問答內容可持續正確且一致。

Acceptance Criteria

  1. 後台支援 FAQ 新增、編輯、下架、搜尋。
  2. 每次更新需保留版本、更新時間、操作人。
  3. 權限至少分為管理員/編輯者，未授權者不可發布。
  4. 單筆 FAQ 更新到可查詢狀態不超過 5 分鐘。
PB-003｜截止日提醒與待辦建立【P0】

User Story

As a 忙碌學生,
I want 把問答中的期限一鍵設為提醒,
so that 我不會錯過重要時程。

Acceptance Criteria

  1. 回答中含日期/截止資訊時，可一鍵建立提醒。
  2. 提醒可設定時間、頻率、通知方式（App 或 Email）。
  3. 已建立提醒可在「我的待辦」查看、編輯、刪除。
  4. 到期前依設定發送通知，失敗需有重試與紀錄。
PB-004｜回答評分與錯答回報流程【P0】

User Story

As a 使用者,
I want 對回答評分並回報錯誤,
so that 系統能更快修正並提高準確率。

Acceptance Criteria

  1. 每則回答提供「有幫助/沒幫助」評分。
  2. 「沒幫助」可填寫錯誤類型與補充說明。
  3. 管理端可查看回報清單與處理狀態（已受理/修正中/已完成）。
  4. 回報案件需可追蹤處理時間與結果備註。
PB-005｜個人化首頁（身分/年級/系所）【P1】

User Story

As a 學生,
I want 看到符合我身分的快捷入口,
so that 我能更快找到需要資訊。

Acceptance Criteria

  1. 首次登入可設定身分、年級、系所。
  2. 首頁依設定顯示對應熱門問題與快捷功能。
  3. 使用者可隨時修改個人化偏好。
  4. 未設定時提供通用首頁與引導設定。
PB-006｜回答轉任務清單（To-do）【P1】

User Story

As a 使用者,
I want 將回答步驟轉為可勾選任務,
so that 我可以按步驟完成流程。

Acceptance Criteria

  1. 回答可一鍵轉為任務清單（含預設步驟）。
  2. 任務支援勾選完成、調整順序、加註備註。
  3. 任務可關聯提醒時間與到期日。
  4. 任務完成率可被統計（個人層級）。
PB-007｜行政儀表板（熱門問題/錯答熱點）【P1】

User Story

As an 行政管理者,
I want 查看熱門問題與錯答熱點,
so that 我能優先優化公告與流程。

Acceptance Criteria

  1. 儀表板顯示 Top 問題、趨勢、時段分布。
  2. 可查看錯答回報量與類型分布。
  3. 支援依單位/期間篩選並匯出 CSV。
  4. 數據每日自動更新，異常時顯示告警。
PB-008｜多輪情境引導（新生報到/畢業檢核）【P2】

User Story

As a 新生或畢業生,
I want 被引導完成多步驟流程,
so that 我不會漏掉必要事項。

Acceptance Criteria

  1. 提供情境流程模板（新生報到、畢業檢核）。
  2. 每步驟含條件、說明、連結與完成標記。
  3. 可中斷後續接續，保留進度。
PB-009｜跨系統進度整合（選課/申請）【P2】

User Story

As a 使用者,
I want 在同一介面看申請進度,
so that 我不需來回切換多系統。

Acceptance Criteria

  1. 可顯示主要流程狀態（已送出/審核中/完成/退件）。
  2. 異常狀態需提供下一步建議與窗口。
  3. 整合失敗時提供明確錯誤訊息與重試機制。
PB-010｜旺季智能分流（模板/人工）【P2】

User Story

As an 行政人員,
I want 在高峰期自動分流問題,
so that 服務品質可維持穩定。

Acceptance Criteria

  1. 可設定旺季期間與分流規則。
  2. 高重複問題先走模板回覆，複雜問題轉人工。
  3. 需有分流命中率與人工接手率報表。

MVP 上線範圍與 Definition of Done

三、MVP 上線範圍（建議）

  • 必含：PB-001 ~ PB-004（P0 全部）
  • 可加值：PB-005 或 PB-007（二選一）
  • 延期：PB-008 ~ PB-010（P2）

四、Definition of Done（DoD）

  1. 通過功能測試與 UAT。
  2. 有基本操作紀錄與錯誤日誌。
  3. 關鍵指標可被追蹤（FTR、TTA、錯答處理時效）。
  4. 文件完成：使用說明、管理手冊、變更紀錄。
